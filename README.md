# K210-MicroPython-Determinism-Benchmark
Benchmarking execution-time determinism of MicroPython workloads on a K210 RISC-V microcontroller using different algorithms. 
Methodology: 
A benchmarking script was developed to quantify the effect of MicroPython's memory management on execution-time determinism. Five well known algorithms were selected: Bubble Sort, Matrix Multiplication (8x8), Iterative Fibonacci, 4-Queens, and List Updating, as well as a supplemental algorithm: Dynamic List Updating. These algorithms were specifically chosen to evaluate different computational workloads, ranging from CPU-bound computation to heap-intensive memory allocation. Each of the algorithms were tested with the Garbage Collector (GC) enabled and disabled to provide a comparative analysis of latency jitter. 
To minimize external influences, the system was isolated from external peripherals and background interruptions. Additionally, a null test was performed to ensure the computer’s stability. Also, all benchmarking was performed on the K210 Developer Kit (RISC-V).  
The script body was run n = 1000 iterations with 5 trials to ensure consistency. After each trial, the K210 board was subject to a physical reset to ensure similar starting conditions for each trial.  Each algorithm was also subject to 100 warm-up iterations prior to data collection to account for irregular start effects. 
Benchmarking scripts were developed and uploaded using the MaixPy IDE. All experiments were executed on a Yahboom K210 Developer Kit running the CanMV MicroPython firmware (v2.0.6, build dated March 17, 2023).  
The Kendryte K210 features a dual-core 64-bit RISC-V architecture with a standard operating frequency of up to 400 MHz. Although higher frequencies (600 MHz) may be achievable through overclocking configurations, no overclocking was applied in this study (8). Inspection of the MicroPython board module on the CanMV-Yahboom K210 platform showed available attributes limited to configuration and hardware metadata (config, board_info, tmp). No interface for runtime CPU frequency or clock configuration was exposed at the MicroPython level. Therefore, system frequency was treated as a fixed, firmware-controlled parameter across all experiments. 
time.ticks_us from MicroPython’s time module was used to calculate execution times per GC state and algorithm across the 5 trials. The function returns an increasing µs-based counter, making it suitable to measure elapsed execution intervals without relying on adjustable system clock values (3). Accordingly, ticks_us was selected to capture the time variation, including the potential latency changes caused by the Garbage Collector. 
A comparative test was implemented to evaluate the effect of enabling and disabling MicroPython’s automatic memory management. Each algorithm was executed with the garbage collector enabled and disabled. When the Garbage Collector was enabled, CanMV’s runtime managed the heap allocation to reflect standard development conditions. After, the GC was disabled by gc.disable(), to stop any automatic memory management. Comparing these configurations isolates the latency introduced by garbage collection.  
Because the recursive implementations of 4-Queens exceeded the board's memory constraints, the code was changed from traditional recursion to an iterative while loop to optimize the code so it can run without memory errors.  
Statistical analysis of the output included the Coefficient of Variation (CV) for relative dispersion, mean to represent the average execution time per algorithm, standard deviation to measure the spread of execution times around the mean, high percentile analysis (p95, p99) to quantify tail latency jitter (4). Outliers were detected by the Median Absolute Deviation (MAD), defined as median(|$x_i$-median(X)|) (Eq1), with a threshold of (3 * MAD * 1.4826)+10^-6 (Eq2)(5). These equations provide a measurable figure to evaluate non-determinism, and outliers were counted and recorded for each algorithm in both GC configurations
All raw data points for each of the 6 algorithms in both GC states across 5 trials were then aggregated to compute the final performance metrics.
To evaluate heap allocation behavior, the Dynamic List Creation workload was added as a supplemental benchmark. During development, it was found that the repeated allocation of nested list objects while the GC was disabled consistently resulted in a MemoryError immediately after the GC Disabled benchmark began. This behavior is consistent with MicroPython’s documented heap objects from being reclaimed unless the gc.collect() is invoked manually(6). 

How To Get Started:
Connect your K210 Board into your computer with a USB-C Cable. Follow instructions on the offical YahBoom K210 Developer Kit page to install the COM port so the board can get recognized. Then, install MaixPy IDE and connect your board to the IDE to start experimenting!
